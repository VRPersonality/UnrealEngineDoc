36/**
37 * A tool activity is a sort of "sub-tool" used to break apart functionality in tools that
38 * provide support for different multi-interaction subtasks. It is meant to limit the sprawl
39 * of switch statements in the base tool, to allow subtasks to be designed similar to how a
40 * tool might be designed, and to ease extendability.
41 *
42 * An activity has the following expectations:
43 * - Setup() is called in host tool setup and Shutdown() is called in host tool Shutdown()
44 * - Start() is called to start the activity (such as when user clicks a button).
45 * - If the activity returns a result of EStartResult::ActivityRunning on Start(), it will expect
46 * Render() and Tick() calls from the host until either (a)- the host calls End() on the activity,
47 * or (b)- the activity reaches a stopping point itself and calls NotifyActivitySelfEnded() on the
48 * host. The activity should not require Render() and Tick() if it is not running.
49 *
50 * Compared to a UInteractionMechanic, a tool activity:
51 * - Expects that it is the main consumer of input, i.e. takes over the tool. Mechanics, by contrast,
52 * are currently often used together with other mechanics, or to support main tool functionality.
53 * - Should not require the hosting tool to have specific knowledge about it or be heavily involved.
54 * Mechanics, by contrast, currently often require tools to use various mechanic-specific getters/setters
55 * during the tool.
56 *
57 * Passing data back and forth can be done either by letting a tool activity use a specific
58 * context object that the tool can prep in the context store, or by requiring the host to
59 * implement specific interfaces (that the activity can check for in Setup()).
60 */
