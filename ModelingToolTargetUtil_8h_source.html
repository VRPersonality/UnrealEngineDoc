23//
24// UE::ToolTarget:: namespace contains utility/helper functions for interacting with UToolTargets.
25// Generally these are meant to be used by UInteractiveTools to handle standard tasks that would
26// otherwise require each Tool to figure out things like which ToolTargetInterface to cast to, etc.
27// Using these functions ideally avoids all the boilerplate inherent in the ToolTarget system.
28//
29// However, the cost is that it is not necessarily the most efficient, as each one of these functions
30// may potentially do many repeated Cast<>'s internally. So, use sparingly, or cache the outputs.
31//

278 /**
279 * Internal path for use by tool targets to invoke PostEditChange with conditional capture for undo
280 * transactions. This capture is controlled by the CVar 'modeling.CapturePostEditChangeInTransactions'.
281 *
282 * PostEditChange can include arbitrary user code which can lead to a tool transaction capturing data
283 * beyond the modifications the tool has made, opening up the risk of various side effects. It is
284 * recommended that PostEditChange is not included in a transaction scope since PostEditUndo will
285 * reinvoke PostEditChange and provide the user code the opportunity to reapply its changes in response
286 * to the data modifications being made.
287 *
288 * That said, not all user code in PostEditChange is well behaved and deterministic so we provide the CVar
289 * to toggle this behavior.
