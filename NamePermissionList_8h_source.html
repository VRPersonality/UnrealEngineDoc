126// Result of non-exact filtering on path prefixes
127enum class EPathPermissionPrefixResult
128{
129 // The query against the list failed because of one of the following:
130 // * There was an explicit allow-list and none of the entries was a parent path of the query path
131 // (or a child of the query path when bAllowParentPaths = true)
132 // * There were deny-list entries and one of the entries was a parent path of the query path
133 Fail,
134 // The query against the list failed and either:
135 // * All paths are denied, so queries for child paths will also fail
136 // * The query failed on a deny-list entry, so child paths will fail on this same entry
137 // * There is an explicit allow-list and there are no entries with more components than the query path,
138 // so children can never pass.
139 FailRecursive,
140 // The query against the list succeeded, but queries for child paths may fail because there are longer
141 // paths in the deny-list with more components
142 Pass,
143 // The query against the list succeeded and queries for child paths will all succeed no deny-list entry can
144 // possibly fail them
145 PassRecursive,
146};

148/**
149 * Set of paths that are allowd and/or denied for certain use cases.
150 * A permission list may contain
151 * - Blanket denial
152 * - Specifically denied paths
153 * - Specifically allowed paths
154 * In decreasing order of priority. When performing prefix checks, if a a path matches a denied path, it cannot be
155 * allowed again by a more specific allowed path. If any paths are specifically allowed, paths which do NOT match
156 * something in the allow list are implicitly denied.

190 /**
191 * Checks the given path against the restrictions and return whether it's possible for any child paths to succeed or
192 * fail as well. Returning PassRecursive or FailRecursive guarantees that no child paths of the queried path can
193 * fail or pass the filter respectively. Returning Pass or Fail does not guarantee that there is some path which
194 * fails or passes the filter respectively.
195 *
196 * Examples:
197 *
198 * Given no deny or allow lists:
199 * Inputs:
200 * / -> PassRecursive, because no paths can fail to match the allow list or match the deny list.
201 *
202 * Given a deny-list entry:
203 * Allow: empty
204 * Deny: /Secret
205 * Inputs:
206 * / -> Pass, because some children of this path may be denied.
207 * /Secret -> FailRecursive, because this path is denied and all children will also be denied.
208 * /Public -> PassRecursive, because this path is not demied and no children can be denied.
209 *
210 * Given an allow-list entry:
211 * Allow: /JustThis
212 * Deny: empty
213 * Inputs:
214 * / -> Fail
215 * /JustThis -> PassRecursive
216 * /SomethingElse -> FailRecursive
217 *
218 * Given both allow and deny-lists:
219 * Allow: /Stuff
220 * Deny: /Stuff/Secret
221 * Inputs:
222 * / -> Fail
223 * /Stuff -> Pass
224 * /Stuff/Secret -> Fail
225 * /Stuff/Public -> PassRecursive
226 */
227 CORE_API EPathPermissionPrefixResult PassesStartsWithFilterRecursive(const FStringView Item,
228 const bool bAllowParentPaths = false) const;
