31/**
32 *****************************************************************************************
33 * NetDrivers, NetConnections, and Channels
34 *****************************************************************************************
35 *
36 *
37 * UNetDrivers are responsible for managing sets of UNetConnections, and data that can be shared between them.
38 * There is typically a relatively small number of UNetDrivers for a given game. These may include:
39 * - The Game NetDriver, responsible for standard game network traffic.
40 * - The Demo NetDriver, responsible for recording or playing back previously recorded game data. This is how Replays work.
41 * - The Beacon NetDriver, responsible for network traffic that falls outside of "normal" gameplay traffic.
42 *
43 * Custom NetDrivers can also be implemented by games or applications and used.
44 * NetConnections represent individual clients that are connected to a game (or more generally, to a NetDriver).
45 *
46 * End point data isn't directly handled by NetConnections. Instead NetConnections will route data to Channels.
47 * Every NetConnection will have its own set of channels.
48 *
49 * Common types of channels:
50 *
51 * - A Control Channel is used to send information regarding state of a connection (whether or not the connection should close, etc.)
52 * - A Voice Channel may be used to send voice data between client and server.
53 * - A Unique Actor Channel will exist for every Actor replicated from the server to the client.
54 *
55 * Custom Channels can also be created and used for specialized purposes (although, this isn't very common).
56 *
57 *
58 *****************************************************************************************
59 * Game Net Drivers, Net Connections, and Channels
60 *****************************************************************************************
61 *
62 *
63 * Under normal circumstances, there will exist only a single NetDriver (created on Client and Server) for "standard" game traffic and connections.
64 *
65 * The Server NetDriver will maintain a list of NetConnections, each representing a player that is in the game. It is responsible for
66 * replicating Actor Data.
67 *
68 * Client NetDrivers will have a single NetConnection representing the connection to the Server.
69 *
70 * On both Server and Client, the NetDriver is responsible for receiving Packets from the network and passing those to the appropriate
71 * NetConnection (and establishing new NetConnections when necessary).
72 *
73 *
74 *****************************************************************************************
75 *****************************************************************************************
76 *****************************************************************************************
77 * Initiating Connections / Handshaking Flow.
78 *****************************************************************************************
79 *****************************************************************************************
80 *****************************************************************************************
81 *
82 *
83 * UIpNetDriver and UIpConnection (or derived classes) are the engine defaults for almost every platform, and everything below
84 * describes how they establish and manage connections. These processes can differ between implementations of NetDriver, however.
85 *
86 * Both Server and Clients will have have their own NetDrivers, and all UE Replicated Game traffic will be sent by or Received
87 * from the IpNetDriver. This traffic also includes logic for establishing connections, and re-establishing connections when
88 * something goes wrong.
89 *
90 * Handshaking is split across a couple of different places: NetDriver, PendingNetGame, World, PacketHandlers, and maybe others.
91 * The split is due to having separate needs, things such as: determining whether or not an incoming connection is sending data in "UE-Protocol",
92 * determining whether or not an address appears to be malicious, whether or not a given client has the correct version of a game, etc.
93 *
94 *
95 *****************************************************************************************
96 * Startup and Handshaking
97 *****************************************************************************************
98 *
99 *
100 * Whenenever a server Loads a map (via UEngine::LoadMap), we will make a call into UWorld::Listen.
101 * That code is responsible for creating the main Game Net Driver, parsing out settings, and calling UNetDriver::InitListen.
102 * Ultimately, that code will be responsible for figuring out what how exactly we listen for client connections.
103 * For example, in IpNetDriver, that is where we determine the IP / Port we will bind to by calls to our configured Socket Subsystem
104 * (see ISocketSubsystem::GetLocalBindAddresses and ISocketSubsystem::BindNextPort).
105 *
106 * Once the server is listening, it's ready to start accepting client connections.
107 *
108 * Whenever a client wants to Join a server, they will first establish a new UPendingNetGame in UEngine::Browse with the server's IP.
109 * UPendingNetGame::Initialize and UPendingNetGame::InitNetDriver are responsible for initializing settings and setting up the NetDriver respectively.
110 * Clients will immediately setup a UNetConnection for the server as a part of this initialization, and will start sending data to the server on that connection,
111 * initiating the handshaking process.
112 *
113 * On both Clients and Server, UNetDriver::TickDispatch is typically responsible for receiving network data.
114 * Typically, when we receive a packet, we inspect its address and see whether or not it's from a connection we already know about.
115 * We determine whether or not we've established a connection for a given source address by simply keeping a map from FInternetAddr to UNetConnection.
116 *
117 * If a packet is from a connection that's already established, we pass the packet along to the connection via UNetConnection::ReceivedRawPacket.
118 * If a packet is not from a connection that's already established, we treat is as "connectionless" and begin the handshaking process.
119 *
120 * See StatelessConnectionHandlerComponent.cpp for details on how this handshaking works.
121 *
122 *
123 *****************************************************************************************
124 * UWorld / UPendingNetGame / AGameModeBase Startup and Handshaking
125 *****************************************************************************************
126 *
127 *
128 * After the UNetDriver and UNetConnection have completed their handshaking process on Client and Server,
129 * UPendingNetGame::SendInitialJoin will be called on the Client to kick off game level handshaking.
130 *
131 * Game Level Handshaking is done through a more structured and involved set of FNetControlMessages.
132 * The full set of control messages can be found in DataChannel.h.
133 *
134 * Most of the work for handling these control messages are done either in UWorld::NotifyControlMessage,
135 * and UPendingNetGame::NotifyControlMessage. Briefly, the flow looks like this:
136 *
137 * Client's UPendingNetGame::SendInitialJoin sends NMT_Hello.
138 *
139 * Server's UWorld::NotifyControlMessage receives NMT_Hello, sends NMT_Challenge.
140 *
141 * Client's UPendingNetGame::NotifyControlMessage receives NMT_Challenge, and sends back data in NMT_Login.
142 *
143 * Server's UWorld::NotifyControlMessage receives NMT_Login, verifies challenge data, and then calls AGameModeBase::PreLogin.
144 * If PreLogin doesn't report any errors, Server calls UWorld::WelcomePlayer, which call AGameModeBase::GameWelcomePlayer,
145 * and send NMT_Welcome with map information.
146 *
147 * Client's UPendingNetGame::NotifyControlMessage receives NMT_Welcome, reads the map info (so it can start loading later),
148 * and sends an NMT_NetSpeed message with the configured Net Speed of the client.
149 *
150 * Server's UWorld::NotifyControlMessage receives NMT_NetSpeed, and adjusts the connections Net Speed appropriately.
151 *
152 * At this point, the handshaking is considered to be complete, and the player is fully connected to the game.
153 * Depending on how long it takes to load the map, the client could still receive some non-handshake control messages
154 * on UPendingNetGame before control transitions to UWorld.
155 *
156 * There are also additional steps for handling Encryption when desired.
157 *
158 *
159 *****************************************************************************************
160 * Reestablishing Lost Connections
161 *****************************************************************************************
162 *
163 *
164 * Throughout the course of a game, it's possible for connections to be lost for a number of reasons.
165 * Internet could drop out, users could switch from LTE to WIFI, they could leave a game, etc.
166 *
167 * If the server initiated one of these disconnects, or is otherwise aware of it (due to a timeout or error),
168 * then the disconnect will be handled by closing the UNetConnection and notifying the game.
169 * At that point, it's up to a game to decide whether or not they support Join In Progress or Rejoins.
170 * If the game does support it, we will completely restart the handshaking flow as above.
171 *
172 * If something just briefly interrupts the client's connection, but the server is never made aware,
173 * then the engine / game will typically recover automatically (albeit with some packet loss / lag spike).
174 *
175 * However, if the Client's IP Address or Port change for any reason, but the server isn't aware of this,
176 * then we will begin a recovery process by redoing the low level handshake. In this case, game code
177 * will not be alerted.
178 *
179 * This process is covered in StatlessConnectionHandlerComponent.cpp.
180 *
181 *
182 *****************************************************************************************
183 *****************************************************************************************
184 *****************************************************************************************
185 * Data Transmission
186 *****************************************************************************************
187 *****************************************************************************************
188 *****************************************************************************************
189 *
190 *
191 * Game NetConnections and NetDrivers are generally agnostic to the underlying communication method / technology used.
192 * That is is left up to subclasses to decide (classes such as UIpConnection / UIpNetDriver or UWebSocketConnection / UWebSocketNetDriver).
193 *
194 * Instead, UNetDriver and UNetConnection work with Packets and Bunches.
195 *
196 * Packets are blobs of data that are sent between pairs of NetConnections on Host and Client.
197 * Packets consist of meta data about the packet (such as header information and acknowledgments), and Bunches.
198 *
199 * Bunches are blobs of data that are sent between pairs of Channels on Host and Client.
200 * When a Connection receives a Packet, that packet will be disassembled into individual bunches.
201 * Those bunches are then passed along to individual Channels to be processed further.
202 *
203 * A Packet may contain no bunches, a single bunch, or multiple bunches.
204 * Because size limits for bunches may be larger than the size limits of a single packet, UE supports
205 * the notion of partial bunches.
206 *
207 * When a bunch is too large, before transmission we will slice it into a number of smaller bunches.
208 * these bunches will be flagged as PartialInitial, Partial, or PartialFinal. Using this information,
209 * we can reassemble the bunches on the receiving end.
210 *
211 * Example: Client RPC to Server.
212 * - Client makes a call to Server_RPC.
213 * - That request is forwarded (via NetDriver and NetConnection) to the Actor Channel that owns the Actor on which the RPC was called.
214 * - The Actor Channel will serialize the RPC Identifier and parameters into a Bunch. The Bunch will also contain the ID of its Actor Channel.
215 * - The Actor Channel will then request the NetConnection send the Bunch.
216 * - Later, the NetConnection will assemble this (and other) data into a Packet which it will send to the server.
217 * - On the Server, the Packet will be received by the NetDriver.
218 * - The NetDriver will inspect the Address that sent the Packet, and hand the Packet over to the appropriate NetConnection.
219 * - The NetConnection will disassemble the Packet into its Bunches (one by one).
220 * - The NetConnection will use the Channel ID on the bunch to Route the bunch to the corresponding Actor Channel.
221 * - The ActorChannel will them disassemble the bunch, see it contains RPC data, and use the RPC ID and serialized parameters
222 * to call the appropriate function on the Actor.
223 *
224 *
225 *****************************************************************************************
226 * Reliability and Retransmission
227 *****************************************************************************************
228 *
229 *
230 * UE Networking typically assumes reliability isn't guaranteed by the underlying network protocol.
231 * Instead, it implements its own reliability and retransmission of both packets and bunches.
232 *
233 * When a NetConnection is established, it will establish a Sequence Number for its packets and bunches.
234 * These can either be fixed, or randomized (when randomized, the sequence will be sent by the server).
235 *
236 * The packet number is per NetConnection, incremented for every packet sent, every packet will include its packet number,
237 * and we will never retransmit a packet with the same packet number.
238 *
239 * The bunch number is per Channel, incremented for every **reliable** bunch sent, and every **reliable** bunch will
240 * include its bunch number. Unlike packets, though, exact (reliable) bunches may be retransmitted. This means we
241 * will resend bunches with the same bunch number.
242 *
243 * Note, throughout the code what are described above as both bunch numbers and packet numbers are commonly referred to
244 * just as sequence numbers. We make the distinction here for clearer understanding.
245 *
246 * --- Detecting Incoming Dropped Packets ---
247 *
248 *
249 * By assigning packet numbers, we can easily detect when incoming packets are lost.
250 * This is done simply by taking the difference between the last successfully received packet number, and the
251 * packet number of the current packet being processed.
252 *
253 * Under good conditions, all packets will be received in the order they are sent. This means that the difference will
254 * be +1.
255 *
256 * If the difference is greater than 1, that indicates that we missed some packets. We will just
257 * assume that the missing packets were dropped, but consider the current packet to have been successfully received,
258 * and use its number going forward.
259 *
260 * If the difference is negative (or 0), that indicates that we either received some packets out of order, or an external
261 * service is trying to resend data to us (remember, the engine will not reuse sequence numbers).
262 *
263 * In either case, the engine will typically ignore the missing or invalid packets, and will not send ACKs for them.
264 *
265 * We do have methods for "fixing" out of order packets that are received on the same frame.
266 * When enabled, if we detect missing packets (difference > 1), we won't process the current packet immediately.
267 * Instead, it will add it to a queue. The next time we receive a packet successfully (difference == 1), we will
268 * see if the head of our queue is properly ordered. If so, we will process it, otherwise we will continue
269 * receiving packets.
270 *
271 * Once we've read all packets that are currently available, we will flush this queue processing any remaining packets.
272 * Anything that's missing at this point will be assumed to have been dropped.
273 *
274 * Every packet successfully received will have its packet number sent back to the sender as an acknowledgment (ACK).
275 *
276 *
277 * --- Detecting Outgoing Dropped Packets ---
278 *
279 *
280 * As mentioned above, whenever a packet is received successfully the recipient will send back an ACK.
281 * These ACKs will contain the packet numbers of successfully received packets, in sequence order.
282 *
283 * Similar to how the recipient tracks the packet number, the sender will track the highest ACKed packet number.
284 *
285 * When ACKs are being processed, any ACK below our last received ACK is ignored and any gaps in packet
286 * numbers are considered Not Acknowledged (NAKed).
287 *
288 * It is the sender's responsibility to handle these ACKs and NAKs and resend any missing data.
289 * The new data will be added to new outgoing packets (again, we will not resend packets we've already sent,
290 * or reuse packet sequence numbers).
291 *
292 *
293 * --- Resending Missing Data ---
294 *
295 *
296 * As mentioned above, packets alone don't contain useful game data. Instead, it's the bunches that comprise them
297 * that have meaningful data.
298 *
299 * Bunches can either be marked as Reliable or Unreliable.
300 *
301 * The engine will make no attempt at resending unreliable bunches if they are dropped. Therefore, if bunches
302 * are marked unreliable, the game / engine should be able to continue without them, or external retry
303 * mechanisms must be put in place, or the data must be sent redundantly. Therefore, everything below only
304 * applies to reliable bunches.
305 *
306 * However, the engine will attempt to resend reliable bunches. Whenever a reliable bunch is sent, it will
307 * be added to a list of un-ACKed reliable bunches. If we receive a NAK for a packet that contained the bunch,
308 * the engine will retransmit an exact copy of that bunch. Note, because bunches may be partial, dropping even
309 * a single partial bunch will result in retransmission of the entire bunch. When all packets containing a bunch
310 * have been ACKed, we will remove it from the list.
311 *
312 * Similar to packets, we will compare the bunch number for received reliable bunches to the last successfully
313 * received bunch. If we detect that the difference is negative, we simply ignore the bunch. If the difference
314 * is greater than one, we will assume we missed a bunch. Unlike packet handling, we will not discard this data.
315 * Instead, we will queue the bunch and pause processing of **any** bunches, reliable or unreliable.
316 * Processing will not be resumed until we detect have received the missing bunches, at which point we will process
317 * them, and then start processing our queued bunches.
318 * Any new bunches that are received while waiting for the missing bunches, or while we still have any bunches in our
319 * queue, will be added to the queue instead of being processed immediately.
320 *
321 */
