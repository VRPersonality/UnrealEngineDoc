21/**
22 * Versioning
23 *
24 * We now have two enums for global versioning, EUnrealEngineObjectUE5Version and EUnrealEngineObjectUE4Version
25 * which can be updated independently of each other. This allows us to potentially make version changed in the UE4
26 * code base in the future and still have packages saved with those changes to be compatible with the UE5 code base.
27 *
28 * The struct FPackageFileVersion should be used to contain the version number rather than storing the UE4/UE5 specific
29 * version numbers. It provides overloads for testing against both version enum types for convenience.
30 *
31 * Note that when reporting versioning problems we often use FPackageFileVersion::ToValue which returns the highest
32 * version valid in the structure. This allows us to keep reporting code simple and hide the fact that there are two
33 * version numbers from end users. However this does mean that if a version mismatch is encountered due to a future
34 * extension of EUnrealEngineObjectUE4Version, then the error message given might look odd. Although given that we are
35 * unlikely to be changing EUnrealEngineObjectUE4Version in the future the risk is deemed worth it.
36 */
