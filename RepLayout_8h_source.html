130 /**
131 * Builds a new ConditionMap given the input RepFlags.
132 * This can be used to determine whether or not a given property should be
133 * considered enabled / disabled based on ELifetimeCondition.
134 */
135 inline TStaticBitArray<COND_Max> BuildConditionMapFromRepFlags(const FReplicationFlags& RepFlags)
136 {
137 TStaticBitArray<COND_Max> ConditionMap;
138
139 // Setup condition map
140 const bool bIsInitial = RepFlags.bNetInitial ? true : false;
141 const bool bIsOwner = RepFlags.bNetOwner ? true : false;
142 const bool bIsSimulated = RepFlags.bNetSimulated ? true : false;
143 const bool bIsPhysics = RepFlags.bRepPhysics ? true : false;
144 const bool bIsReplay = RepFlags.bReplay ? true : false;
145
146 ConditionMap[COND_None] = true;
147 ConditionMap[COND_InitialOnly] = bIsInitial;
148
149 ConditionMap[COND_OwnerOnly] = bIsOwner;
150 ConditionMap[COND_SkipOwner] = !bIsOwner;
151
152 ConditionMap[COND_SimulatedOnly] = bIsSimulated;
153 ConditionMap[COND_SimulatedOnlyNoReplay] = bIsSimulated && !bIsReplay;
154 ConditionMap[COND_AutonomousOnly] = !bIsSimulated;
155
156 ConditionMap[COND_SimulatedOrPhysics] = bIsSimulated || bIsPhysics;
157 ConditionMap[COND_SimulatedOrPhysicsNoReplay] = (bIsSimulated || bIsPhysics) && !bIsReplay;
158
159 ConditionMap[COND_InitialOrOwner] = bIsInitial || bIsOwner;
160 ConditionMap[COND_ReplayOrOwner] = bIsReplay || bIsOwner;
161 ConditionMap[COND_ReplayOnly] = bIsReplay;
162 ConditionMap[COND_SkipReplay] = !bIsReplay;
163
164 ConditionMap[COND_Custom] = true;
165 ConditionMap[COND_Dynamic] = true;
166 ConditionMap[COND_Never] = false;
167
168 return ConditionMap;
169 }

1085/**
1086 * This class holds all replicated properties for a given type (either a UClass, UStruct, or UFunction).
1087 * Helpers functions exist to read, write, and compare property state.
1088 *
1089 * There is only one FRepLayout for a given type, meaning all instances of the type share the FRepState.
1090 *
1091 * COMMANDS:
1092 *
1093 * All Properties in a RepLayout are represented as Layout Commands.
1094 * These commands dictate:
1095 * - What the underlying data type is.
1096 * - How the data is laid out in memory.
1097 * - How the data should be serialized.
1098 * - How the data should be compared (between instances of Objects, Structs, etc.).
1099 * - Whether or not the data should trigger notifications on change (RepNotifies).
1100 * - Whether or not the data is conditional (e.g. may be skipped when sending to some or all connections).
1101 *
1102 * Commands are split into 2 main types: Parent Commands (@see FRepParentCmd) and Child Commands (@see FRepLayoutCmd).
1103 *
1104 * A Parent Command represents a Top Level Property of the type represented by an FRepLayout.
1105 * A Child Command represents any Property (even nested properties).
1106 *
1107 * E.G.,
1108 * Imagine an Object O, with 4 Properties, CA, DA, I, and S.
1109 * CA is a fixed size C-Style array. This will generate 1 Parent Command and 1 Child Command for *each* element in the array.
1110 * DA is a dynamic array (TArray). This will generate only 1 Parent Command and 1 Child Command, both referencing the array.
1111 * Additionally, Child Commands will be added recursively for the element type of the array.
1112 * S is a UStruct.
1113 * All struct types generate 1 Parent Command for the Struct Property. Additionally:
1114 * If the struct has a native NetSerialize method then it will generate 1 Child Command referencing the struct.
1115 * If the struct has a native NetDeltaSerialize method then it will generate no Child Commands.
1116 * All other structs will recursively generate Child Commands for each nested Net Property in the struct.
1117 * Note, in this case there is no Child Command associated with the top level struct property.
1118 * I is an integer (or other supported non-Struct type, or object reference). This will generate 1 Parent Command and 1 Child Command.
1119 *
1120 * CHANGELISTS
1121 *
1122 * Along with Layout Commands that describe the Properties in a type, RepLayout uses changelists to know
1123 * what Properties have changed between frames. @see FRepChangedHistory.
1124 *
1125 * Changelists are arrays of Property Handles that describe what Properties have changed, however they don't
1126 * track the actual values of the Properties.
1127 *
1128 * Changelists can contain "sub-changelists" for arrays. Formally, they can be described as the following grammar:
1129 *
1130 * Terminator ::= 0
1131 * Handle ::= Integer between 1 ~ 65535
1132 * Number ::= Integer between 0 ~ 65535
1133 * Changelist ::= <Terminator> | <Handle><Changelist> | <Handle><Array-Changelist><Changelist>
1134 * Array-Changelist: ::= <Number><Changelist>
1135 *
1136 * An important distinction is that Handles do not have a 1:1 mapping with RepLayoutCommands.
1137 * Handles are 1-based (as opposed to 0-based), and track a relative command index within a single
1138 * level of a changelist. Each Array Command, regardless of the number of child Commands it has,
1139 * will only be count as a single handle in its owning changelist. Each time we recurse into an Array-Changelist,
1140 * our handles restart at 1 for that "depth", and they correspond to the Commands associated with the Array's element type.
1141 *
1142 * In order to generate Changelists, Layout Commands are sequentially applied that compare the values
1143 * of an object's cached state to a object's current state. Any properties that are found to be different
1144 * will have their handle written into the changelist. This means handles within a changelists are
1145 * inherently ordered (with arrays inserted whose Handles are also ordered).
1146 *
1147 * When we want to replicate properties for an object, merge together any outstanding changelists
1148 * and then iterate over it using Layout Commands that serialize the necessary property data.
1149 *
1150 * Receiving is very similar, except the Handles are baked into the serialized data so no
1151 * explicit changelist is required. As each Handle is read, a Layout Command is applied
1152 * that serializes the data from the network bunch and applies it to an object.
1153 *
1154 * RETRIES AND RELIABLES
1155 *
1156 * @FSendingRepState maintains a circular buffer that tracks recently sent Changelists (@FRepChangedHistory).
1157 * These history items track the Changelist alongside the Packet ID that the bunches were sent in.
1158 *
1159 * Once we receive ACKs for all associated packets, the history will be removed from the buffer.
1160 * If NAKs are received for any of the packets, we will merge the changelist into the next set of properties we replicate.
1161 *
1162 * If we receive no NAKs or ACKs for an extended period, to prevent overflows in the history buffer,
1163 * we will merge the entire buffer into a single monolithic changelist which will be sent alongside the next set of properties.
1164 *
1165 * In both cases of NAKs or no response, the merged changelists will be tracked in the latest history item
1166 * alongside with other sent properties.
1167 *
1168 * When "net.PartialBunchReliableThreshold" is non-zero and property data bunches are split into partial bunches above
1169 * the threshold, we will not generate a history item. Instead, we will rely on the reliable bunch framework for resends
1170 * and replication of the Object will be completely paused until the property bunches are acknowledged.
1171 * However, this will not affect other history items since they are still unreliable.
1172 */
