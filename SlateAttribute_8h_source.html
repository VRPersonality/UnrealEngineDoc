23/**
24 * Use TSlateAttribute when it's a SWidget member.
25 * Use TSlateManagedAttribute when it's a member inside an array or other moving structure, and the array is a SWidget member. They can only be moved (they can't be copied). THey consume more memory.
26 * For everything else, use TAttribute.
27 *
28 *
29 *
30 * In Slate, TAttributes are optimized for developer efficiency.
31 * They enable widgets to poll for data instead of requiring the user to manually set the state on widgets.
32 * Attributes generally work well when performance is not a concern but break down when performance is critical (like a game UI).
33 *
34 * The invalidation system allows only widgets that have changed to perform expensive Slate layout.
35 * Bound TAttributes are incompatible with invalidation because we do not know when the data changes.
36 * Additionally TAttributes for common functionality such as visibility are called multiple times per frame and the delegate overhead for that alone is very high.
37 * TAttributes have a high memory overhead and are not cache-friendly.
38 *
39 * TSlateAttribute makes the attribute system viable for invalidation and more performance-friendly while keeping the benefits of attributes intact.
40 * TSlateAttributes are updated once per frame in the Prepass update phase. If the cached value of the TSlateAttribute changes, then it will invalidate the widget.
41 * The member attributes are updated in the order the variables are defined in the SWidget definition (by default).
42 * The managed attributes are updated in a random order (after TSlateAttribute).
43 * The update order of member attributes can be defined/override by setting a Prerequisite (see bellow for an example).
44 * The invalidation reason can be a predicate (see bellow for an example).
45 * The invalidation reason can be override per SWidget. Use override with precaution since it can break the invalidation of widget's parent.
46 * The widget attributes are updated only if the widget is visible/not collapsed.
47 *
48 *
49 * TSlateAttribute is not copyable and can only live inside a SWidget.
50 * For performance reasons, we do not save the extra information that would be needed to be "memory save" in all contexts.
51 * If you create a TSlateAttribute that can be moved, you need to use TSlateManagedAttribute.
52 * TSlateManagedAttribute is as fast as TSlateAttribute but use more memory and is less cache-friendly.
53 * Note, if you use TAttribute to change the state of a SWidget, you need to override bool ComputeVolatility() const.
54 * Note, ComputeVolatility() is not needed for TSlateAttribute and TSlateManagedAttribute
55 *
56 * TSlateAttribute request a SWidget pointer. The "this" pointer should ALWAYS be used.
57 * (The SlateAttribute pointer is saved inside the SlateAttributeMetaData. The widget needs to be aware when the pointer changes.)
58 *
59 *

179/** Base struct of all SlateAttribute type. */
180struct FSlateAttributeBase
181{
182 /**
183 * Not all invalidation is supported by SlateAttribute.
184 * ChildOrder: The update of SlateAttribute is done in the SlatePrepass. We can't add or remove children in SlatePrepass.
185 * AttributeRegistration: In FastPath, the SlateAttribute are updated in a loop. The iterator can't be modified while we are looping.
186 */
187 template<typename T>
188 constexpr static bool IsInvalidateWidgetReasonSupported(T Reason)
189 {
190 return false;
191 }
192 constexpr static bool IsInvalidateWidgetReasonSupported(EInvalidateWidgetReason Reason)
193 {
194 return (Reason & (EInvalidateWidgetReason::ChildOrder | EInvalidateWidgetReason::AttributeRegistration)) == EInvalidateWidgetReason::None;
195 }
196};
