95 // Use this if your cell subtype has any outgoing strong references. It is used by both the
96 // GC system and the abstract visitor to collect strong references.
97 //
98 // Note that this function will run concurrently to the mutator (this function is being called in
99 // some collector thread while the VM's other threads are calling other methods on your object)
100 // and in parallel (the collector will run multiple threads calling this function). However,
101 // you're guaranteed that this function will only be called once per object per collection cycle;
102 // i.e. the GC will never call this function simultaneously for the same object.
103 //
104 // It is defined implicitly by DECLARE_DERIVED_VCPPCLASSINFO, and so you must either implement it
105 // or use the DEFINE_TRIVIAL_VISIT_REFERENCES macro to explicitly define a trivial implementation.

117 //
118 // Note that this function may run concurrently to the mutator, in parallel, or from within the

227 // Inform the GC that this cell is now a key in the following weak map and it keeps the given value alive.
228 // For sound concurrent GC handling of weak maps, both the map and the key must know about the mapping.
229 // Hence, every cell in the VVM has the secret ability to become a weak key. This costs just one bit per
230 // object if unused. It costs horrors and nightmares if used (but at least the terror is O(1)ish).
231 void AddWeakMapping(VCell* Map, VCell* Value);
232 void RemoveWeakMapping(VCell* Map);
233
234 // This function is a test-only function because it has a very limited kind of meaning. Requesting the size (or
235 // checking the emptiness) of a weap map (including a "transposed" weak map, like the weak key map) gives a kind
236 // of upper bound: it means that the map has at most this many entries. But we cannot tell you which of those
237 // entries are real. When you query them, you are likely to find fewer entries.
