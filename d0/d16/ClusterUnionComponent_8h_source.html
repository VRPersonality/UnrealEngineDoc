186/**
187 * This does the bulk of the work exposing a physics cluster union to the game thread.
188 * This component needs to be a primitive component primarily because of how physics
189 * proxies need to be registered with the solver with an association with a primitive component.
190 * This component can be used as part of AClusterUnionActor or on its own as its list of
191 * clustered components/actors can be specified dynamically at runtime and/or statically
192 * on asset creation.
193 *
194 * The cluster union component needs to not only maintain a game thread representation of what's happening on the
195 * physics thread but it also needs to make sure this data gets replicated to every client. A general model of how
196 * the data flow happens is as follows:
197 *
198 * [Server GT Command] -> [Server PT Command] -> [Server Modifies PT Data] -> [Server Sync PT Data back to GT Data].
199 *
200 * This enables GT control over what happens to the cluster union BUT ALSO maintains a physics-first approach
201 * to the cluster union where a physics event can possibly cause the cluster union to break.
202 *
203 * The GT data is replicated from the server to the clients either via the FClusterUnionReplicatedData on the cluster union component
204 * or per-child component data is replicated via the UClusterUnionReplicatedProxyComponent. Generally, the same flow is
205 * replicated on the client. The only exception is for replicating the X/R/V/W properties on the cluster union particle which does
206 * a GT -> PT data sync. There's no particula reason this happens...it just mirrors the single particle physics proxy here.
207 *
208 */
