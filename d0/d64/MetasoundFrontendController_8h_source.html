23/* Metasound Controllers and Handles provide a object oriented interface for manipulating Metasound Documents.
24 *
25 * Each controller interface is associated with a single Metasound entity such as
26 * Document, Graph, Node, Input or Output. Access to these entities generally begins
27 * with the DocumentController which provides access to Graphs. Graphs provide access
28 * to Nodes and Nodes provide access to Inputs and Outputs.
29 *
30 * A "Handle" is simply a TSharedRef<> of a controller.
31 *
32 * In general, the workflow for editing a Metasound graph will be:
33 * 1) Load or create a metasound asset.
34 * 2) Call UMetaSoundPatch::GetDocumentHandle() to get a handle to the document for that asset.
35 *
36 * Typically the workflow for creating a Metasound subgraph will be
37 * 1) Get a Metasound::Frontend::FGraphHandle (typically through the two steps described in the first paragraph)
38 * 2) Build a FMetasoundFrontendClassMetadata struct with whatever name/author/description you want to give the subgraph,
39 * 3) call Metasound::Frontend::FGraphHandle::CreateEmptySubgraphNode, which will return that subgraph as it as a FNodeHandle for that subgraph in the current graph.
40 * 4) class Metasound::Frontend::FNodeHandle::AsGraph which provides access to edit the subgraphs internal structure as well as externally facing inputs and outputs.
41 *
42 * General note- these apis are NOT thread safe.
43 * Make sure that any FDocumentHandle, FGraphHandle, FNodeHandle, FInputHandle and FOutputHandle that access similar data are called on the same thread.
44 */
