19/**
20 * AbilityTasks are small, self contained operations that can be performed while executing an ability.
21 * They are latent/asynchronous is nature. They will generally follow the pattern of 'start something and wait until it is finished or interrupted'
22 *
23 * We have code in K2Node_LatentAbilityCall to make using these in blueprints streamlined. The best way to become familiar with AbilityTasks is to
24 * look at existing tasks like UAbilityTask_WaitOverlap (very simple) and UAbilityTask_WaitTargetData (much more complex).
25 *
26 * These are the basic requirements for using an ability task:
27 *
28 * 1) Define dynamic multicast, BlueprintAssignable delegates in your AbilityTask. These are the OUTPUTs of your task. When these delegates fire,
29 * execution resumes in the calling blueprints.
30 *
31 * 2) Your inputs are defined by a static factory function which will instantiate an instance of your task. The parameters of this function define
32 * the INPUTs into your task. All the factory function should do is instantiate your task and possibly set starting parameters. It should NOT invoke
33 * any of the callback delegates!
34 *
35 * 3) Implement a Activate() function (defined here in base class). This function should actually start/execute your task logic. It is safe to invoke
36 * callback delegates here.
37 *
38 *
39 * This is all you need for basic AbilityTasks.
40 *
41 *
42 * CheckList:
43 * -Override ::OnDestroy() and unregister any callbacks that the task registered. Call Super::EndTask too!
44 * -Implemented an Activate function which truly 'starts' the task. Do not 'start' the task in your static factory function!
45 *
46 *
47 * --------------------------------------
48 *
49 * We have additional support for AbilityTasks that want to spawn actors. Though this could be accomplished in an Activate() function, it would not be
50 * possible to pass in dynamic "ExposeOnSpawn" actor properties. This is a powerful feature of blueprints, in order to support this, you need to implement
51 * a different step 3:
52 *
53 * Instead of an Activate() function, you should implement a BeginSpawningActor() and FinishSpawningActor() function.
54 *
55 * BeginSpawningActor() must take in a TSubclassOf<YourActorClassToSpawn> parameters named 'Class'. It must also have a out reference parameters of type
56 * YourActorClassToSpawn*& named SpawnedActor. This function is allowed to decide whether it wants to spawn the actor or not (useful if wishing to
57 * predicate actor spawning on network authority).
58 *
59 * BeginSpawningActor() can instantiate an actor with SpawnActorDeferred. This is important, otherwise the UCS will run before spawn parameters are set.
60 * BeginSpawningActor() should also set the SpawnedActor parameter to the actor it spawned.
61 *
62 * [Next, the generated byte code will set the expose on spawn parameters to whatever the user has set]
63 *
64 * If you spawned something, FinishSpawningActor() will be called and pass in the same actor that was just spawned. You MUST call ExecuteConstruction + PostActorConstruction
65 * on this actor!
66 *
67 * This is a lot of steps but in general, AbilityTask_SpawnActor() gives a clear, minimal example.
68 *
69 *
70 */

72/**
73 * Latent tasks are waiting on something. This is to differeniate waiting on the user to do something vs waiting on the game to do something.
74 * Tasks start WaitingOnGame, and are set to WaitingOnUser when appropriate (see WaitTargetData, WaitIiputPress, etc)
75 */
76UENUM()
77enum class EAbilityTaskWaitState : uint8
78{
79 /** Task is waiting for the game to do something */
80 WaitingOnGame = 0x01,
81
82 /** Waiting for the user to do something */
83 WaitingOnUser = 0x02,
84
85 /** Waiting on Avatar (Character/Pawn/Actor) to do something (usually something physical in the world, like land, move, etc) */
86 WaitingOnAvatar = 0x04
87};
