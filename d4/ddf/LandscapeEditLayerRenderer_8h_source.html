165/**
166 * Each edit layer render item represents the capabilities of what a given edit layer can render in terms of landscape data : a renderer can provide one or many render items, which contain
167 * the "locality" (what area do I affect?) as well as the "capability" (what target tool type do I affect? what weightmap(s)?) information related to what this layer item can do.
168 * See ILandscapeEditLayerRenderer::GetRenderItems
169 */
170class FEditLayerRenderItem
171{
172public:
173 FEditLayerRenderItem() = delete;

175 : TargetTypeState(InTargetTypeState)
176 , InputWorldArea(InInputWorldArea)
177 , OutputWorldArea(InOutputWorldArea)
178 , bModifyExistingWeightmapsOnly(bInModifyExistingWeightmapsOnly)
179 {}
180
181 const FEditLayerTargetTypeState& GetTargetTypeState() const { return TargetTypeState; }
182
183 const FInputWorldArea& GetInputWorldArea() const { return InputWorldArea; }

185
186 const FOutputWorldArea& GetOutputWorldArea() const { return OutputWorldArea; }
187 void SetOutputWorldArea(const FOutputWorldArea& InOutputWorldArea) { OutputWorldArea = InOutputWorldArea; }
188
189 bool GetModifyExistingWeightmapsOnly() const { return bModifyExistingWeightmapsOnly; }
190
191private:
192
193 /** Target types / weightmaps that this render item writes to */
194 FEditLayerTargetTypeState TargetTypeState;
195
196 /**
197 * Area that this render item needs in order to render properly.
198 * - If Infinite, it is assumed the render item needs the entire loaded landscape to render properly (i.e. it's dependent on all loaded landscape components)
199 * - If Local, it requires a particular component and optionally its immediate neighbors
200 * - If OOBox, then only the landscape components covered by this area will be considered as inputs
201 */
202 FInputWorldArea InputWorldArea;
203
204 /**
205 * Area that this render item writes to.
206 * - If Infinite, the render item writes everywhere (use only if necessary, as it will
207 * - If Local, it requires a particular component and optionally its immediate neighbors
208 * - If OOBox, then only the landscape components covered by this area will be considered as inputs
209 */
210 FOutputWorldArea OutputWorldArea;
211
212 /** Indicates whether this render item actually outputs weightmaps (if false) or only modifies existing ones underneath (i.e. blending-only)*/
213 bool bModifyExistingWeightmapsOnly = false;

267 /**
268 * GetRendererStateInfo retrieves the current state of this renderer (what it can and does render, as well as how to group target layers together), and part of this will then be mutable for the duration of the merge.
269 * The idea is that FEditLayerRendererState's SupportedTargetTypeState tells the capabilities of this renderer, while EnabledTargetTypeState tells what it currently does render.

318 /**
319 * RenderLayer is where the renderer has a chance to render its content and eventually blend it with the merged result of all preceding layers (if ERenderFlags::BlendMode_SeparateBlend is not returned)
320 * It operates on a limited set of components (depending on the size of the render batches) and on a set of target layers (e.g. multiple weightmaps).
321 * It guarantees access to the merged result from preceding layers of each target layer.
322 *
323 * @param RenderParams contains parameters necessary for this render (affected components, merge context, etc.)
324 *
325 * @param RDGBuilderRecorder recorder to append operations to a single FRDGBuilder, if the the recorder in "recording" mode. In "immediate" mode, a render command will be enqueued immediately.
326 * The recorder can be in 2 states, depending on the result from GetRenderFlags
327 * - ERenderFlags::RenderMode_Immediate : the recorder is in immediate mode renderer can enqueue render commands just like any other game thread-based renderer
328 * - ERenderFlags::RenderMode_Recorded : the function runs on the game thread but is *not* meant to enqueue render commands directly. Instead, it registers
329 * consecutive "render commands" via "RDG/render command" accumulated on a FRDGBuilderRecorder. This allows to coalesce several render commands onto the same FRDGBuilder (which is critical
330 * for performance) while still allowing to interleave game thread-based renders (flushing the command recorder, enqueuing the game thread-based render, and starting recording on the same (or a
331 * new) command recorder again...)
332 * TLDR: when using the ERenderFlags::RenderMode_Recorded method, the user should use the command recorder to enqueue lambdas instead if enqueuing render commands directly.
333 * Corollary: Any render command issued by RenderLayer will end up being pushed before the render operations recorded on the RDG command recorder (unless it is flushed), so there's no reason
334 * to actually do it, unless you want those commands to run before the recorded commands
335 *
336 * @return true if anything was rendered
337 */
