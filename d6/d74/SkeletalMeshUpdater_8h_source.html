19/*
20 The Skeletal Mesh Updater is an optimized pipeline for processing batches of skeletal mesh updates in parallel
21 with other scene rendering tasks. Each scene has an instance of this class. The system supports processing
22 commands from multiple channels, where each channel is a separate backend implementation. The resulting work
23 is processed in the same update tasks on the rendering timeline.
24
25 Frontend Usage:
26
27 Mesh objects are registered with the system on the game thread to receive a handle. The handle provides an interface to
28 push updates as well as release the instance:
29
30 FSkeletalMeshUpdater* Updater = Scene->GetSkeletalMeshUpdater();
31
32 // Creates a new handle associated with the object.
33 FSkeletalMeshUpdateHandle Handle = Updater->Create(MyMeshObject);
34
35 // Pushes a new update for the associated object.
36 Handle.Update(MyDynamicData);
37
38 // Releases the object from the updater.
39 Handle.Release();
40
41 Commands are automatically pushed by the system from the game thread to the render thread at sync points prior to rendering.
42 A delegate associated with UE::RenderCommandPipe::FSyncScope is used for this, as it already instruments key sync points.
43 Updates are not actually replayed until the next scene render, so an instance can very well receive multiple update requests
44 and then get removed. Removal is handled safely by releasing all pending updates and unregistering the instance. For multiple
45 updates, all intermediates must be processed back-to-back immediately and only the final update can get batched. This is to
46 simplify the batched path so that it does not have to handle multiple queued up states. In practice this scenario is rare and
47 is not on the performance critical path.
48
49 Backend Usage:
50
51 Updates are processed in stages based on the work that needs to be synced first in the pipeline. Mesh Deformer related work is synced
52 first, followed by skin cache, and finally inline skinning that touches mesh draw commands. A background task is kicked afterwards and this
53 is useful for performing RHI related updates; e.g. filling data into bone buffers or updating vertex factories, since that work does not
54 need to be synced until much later. Ideally only setup work is performed during the foreground stages.
55
56 Implementing a backend requires deriving from TSkeletalMeshUpdatePacket, which is associated with derived FSkeletalMeshObject / FSkeletalMeshDynamicData
57 types. As commands are replayed they are filtered into the user-derived TSkeletalMeshUpdatePacket class using template method overrides. Later the process
58 method associated with each stage is called in order. See the following example code for usage:
59
60 class FMyMeshObject : public FSkeletalMeshObject { ... };
61 class FMyMeshDynamicData : public FSkeletalMeshDynamicData { ... };
62
63 class FMySkeletalMeshUpdatePacket : public TSkeletalMeshUpdatePacket<FMyMeshObject, FMyMeshDynamicData>
64 {
65 public:
66 void Add(MeshObjectType* MeshObject, MeshDynamicDataType* MeshDynamicData)
67 {
68 // Filter the mesh object into the N stages of processing.
69 }
70
71 void UpdateImmediate(FRHICommandList& RHICmdList, MeshObjectType* MeshObject, MeshDynamicDataType* MeshDynamicData)
72 {
73 // Process this update immediately instead. This is how intermediate updates are handled if multiples get queued up.
74 }
75
76 void Process_SkinCache(FRHICommandList& RHICmdList)
77 {
78 // Do setup work associated with the skin cache (create buffers, register skin cache entries, etc).
79 }
80
81 void Process_Background(FRHICommandList& RHICmdList)
82 {
83 // Do update work for RHI resources that were allocated during setup.
84 }
85 }
86
87 // Registers the implementation and sets up a new channel in the updater.
88 REGISTER_SKELETAL_MESH_UPDATE_BACKEND(FMySkeletalMeshUpdatePacket);
89*/
