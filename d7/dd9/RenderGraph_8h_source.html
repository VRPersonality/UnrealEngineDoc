5/** Rendering Dependency Graph framework to setup lambda scopes designed as passes to issue GPU commands to the RHI
6 * through a deferred execution. They are created with FRDGBuilder::AddPass(). When creating a pass,
7 * it needs to have shader parameters. Can be any shader parameters, but the framework is mostly interested about the
8 * render graph resources.
9 *
10 * The structure that hold all the pass parameter should be allocated using FRDGBuilder::AllocParameters() to ensure
11 * correct lifetime due to the lambda's execution being deferred.
12 *
13 * A render graph resource, created with FRDGBuilder::CreateTexture() or FRDGBuilder::CreateBuffer(), only record the
14 * resource descriptor. Allocation will be done by the graph if/when the resource is needed. The graph will track the
15 * resource's lifetime and can free and reuse the memory when the remaining passes no longer reference it.
16 *
17 * All render graph resources used in a pass, must be in the parameters of pass given on the FRDGBuilder::AddPass(). It
18 * allows render graph to know what resource are being used by a given pass.
19 * The resources are only guaranteed to be allocated when executing passes. Therefor accessing them should only done
20 * within the lambda scope of the pass created with FRDGBuilder::AddPass(). Failing to list some resource used by the pass
21 * may result in problems.
22 *
23 * It is important to not reference more graph resources in the parameters than a pass needs, since this artificially
24 * increases the graphs knowledge of that resource's lifetime. This can cause an increase in memory use or prevent passes
25 * from overlapping their execution. An example is ClearUnusedGraphResources() that can automatically null out resource
26 * references unused by a shader for passes that have only on shader. Warnings will be emitted if a resource is not used
27 * in a pass (thanks to FRDGResource::MarkResourceAsUsed())
28 *
29 * The lambda scope of a pass execution may happen any time after FRDGBuilder::AddPass(). For debugging purpose, it may
30 * happen directly in the AddPass() with the immediate mode. When a bug is happening during pass execution, immediate mode
31 * allows you to have the callstack of the pass setup that may have the source cause of the bug. The immediate mode can be
32 * enabled from command line with -rdgimmediate, or with the cvar r.RDG.ImmediateMode=1.
33 *
34 * Pooled managed resource texture FPooledRenderTarget generated by legacy code can be used in render graph by using
35 * FRDGBuilder::RegisterExternalTexture().
36 *
37 * With the knowledge of pass dependencies, the execution may prioritize different hardware goal, such as prioritizing memory
38 * pressure or pass GPU execution concurrency. As such, the execution order of passes is not guaranteed. The execution
39 * order of the pass only guarantees that the execution will perform the work on the intermediary resource exactly as the
40 * immediate mode does on the GPU.
41 *
42 * A render graph pass should not modify the state of a external data structure, as this may lead to edges case according to
43 * the execution order of the pass. Render graph resources that should survive the render graph execution (For instance, a
44 * viewport back buffer, temporal AA history for next frame...) should be extracted using
45 * FRDGBuilder::QueueTextureExtraction(). If a pass is detected not useful to produce any of this resources scheduled for
46 * extraction, or modifying an external texture, this pass may not even be executed with a warning (TODO).
47 *
48 * Unless exception motivated with strong technical reasons (like stereo rendering rendering multiple views at once for VR),
49 * do not bundle multiple work on different resources in the same pass. This will end up creating more dependencies on a
50 * bundle of work that individually may only require a subset of those dependencies. The scheduler may be able to overlap a
51 * part of that with other GPU work. This may also retain allocated transient resources longer, potentially increasing
52 * highest memory pressure peak of your entire frame.
53 *
54 * Although the AddPass() only wants a lambda scope to have deferred execution, it does not mean you nead to write one.
55 * A pass may already be fully implementable using an even simpler utility such as the ones in FComputeShaderUtils
56 * or FPixelShaderUtils.
57 */
