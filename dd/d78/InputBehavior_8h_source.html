223/**
224 * An InputBehavior implements a state machine for a user interaction.
225 * The InputRouter maintains a set of active Behaviors, and when new input
226 * events occur, it calls WantsCapture() to check if the Behavior would like to
227 * begin capturing the applicable input event stream (eg for a mouse, one or both VR controllers, etc).
228 * If the Behavior acquires capture, UpdateCapture() is called until the Behavior
229 * indicates that it wants to release the device, or until the InputRouter force-terminates
230 * the capture via ForceEndCapture().
231 *
232 * For example, something like ButtonSetClickBehavior might work as follows:
233 * - in WantsCapture(), if left mouse is pressed and a button is under cursor, return Begin, otherwise Ignore
234 * - in BeginCapture(), save identifier for button that is under cursor
235 * - in UpdateCapture()
236 * - if left mouse is down, return Continue
237 * - if left mouse is released:
238 * - if saved button is still under cursor, call button.Clicked()
239 * - return End
240 *
241 * Written sufficiently generically, the above Behavior doesn't need to know about buttons,
242 * it just needs to know how to hit-test the clickable object(s). Similarly separate
243 * Behaviors can be written for mouse, VR, touch, gamepad, etc.
244 *
245 * Implementing interactions in this way allows the input handling to be separated from functionality.
246 */
