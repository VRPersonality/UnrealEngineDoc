24/**
25 * An attachment is either null, raw binary, compressed binary, or an object.
26 *
27 * Attachments are always identified by their raw hash, even when stored compressed.
28 *
29 * An attachment is serialized as a sequence of compact binary fields with no name. It is invalid
30 * to serialize a null attachment. A raw binary attachment is serialized as an empty Binary field
31 * when empty, and otherwise as a BinaryAttachment field containing the raw hash, followed by the
32 * raw binary data in a Binary field. A compressed binary attachment is serialized as Binary with
33 * a compressed buffer as the value. An object is serialized as an empty Object field when empty,
34 * and otherwise as an ObjectAttachment field containing the hash, followed by the object data in
35 * an Object field.
36 */

233/**
234 * A package is an object with a tree of referenced attachments.
235 *
236 * A package is a Merkle tree with an object as its root and non-leaf nodes, and either binary or
237 * objects as its leaf nodes. Nodes reference their children from fields of type BinaryAttachment
238 * or ObjectAttachment, which store the raw hash of the referenced attachment.
239 *
240 * It is invalid for a package to include attachments that are not referenced by its object or by
241 * one of its referenced object attachments. This invariant needs to be maintained if attachments
242 * are added explicitly instead of being discovered by the attachment resolver. If any attachment
243 * is not referenced, it may not survive a round-trip through certain storage systems.
244 *
245 * It is valid for a package to exclude referenced attachments, but then it is the responsibility
246 * of the package consumer to have a mechanism for resolving those references when necessary.
247 *
248 * A package is serialized as a sequence of compact binary fields with no name. The object may be
249 * both preceded and followed by attachments. The object is only serialized when it is non-empty,
250 * starting with its hash, in a Hash field, followed by the object, in an Object field. A package
251 * ends with a Null field. The canonical order of components is the object and its hash, followed
252 * by the attachments ordered by hash, followed by a Null field. It is valid for the a package to
253 * have its attachments serialized in any order relative to each other and to the object.
254 */
